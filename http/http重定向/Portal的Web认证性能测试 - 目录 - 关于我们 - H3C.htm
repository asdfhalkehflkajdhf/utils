Portal的Web认证性能测试

发布时间：2013年05月20日

文/曾劼

Portal在英语中有“入口”的意思，一般将Portal认证网站称为门户网站。Portal认证通常也被称为Web认证，在很多场景的网络应用中，都是通过Web进行认证的，比如机场、火车站、咖啡厅等。而Web认证对于网络设备和Portal服务器的压力比客户端认证大很多，如何模拟大量用户进行Web认证，从而验证产品性能、保证产品质量成了一道必须解决的难题。

Portal认证没有一个统一的标准，通常每个设备厂商和运营商都有自己的实现方式。本文以H3C设备为例，介绍如何进行Portal的Web认证性能测试。

如果采用H3C设备进行Portal认证，那么用户有两种方式进行身份验证：一种是使用iNode客户端，另外一种就是通过Web界面。通过Web界面进行身份验证时，用户可以主动访问已知的Portal认证网站，输入用户名和密码进行认证，这种方式被称作主动认证。反之，如果用户试图通过HTTP访问其他外网，将被强制访问Portal认证网站，开始Portal认证过程，这种方式被称作强制认证。

1 Portal Web认证流程分析

首先对Portal Web认证的流程进行简单的分析，整个认证流程如图1所示。

图1 Portal认证流程图

1.1 HTTP访问重定向到Portal认证页面

当用户没有通过验证去访问一个Web站点的时候，设备会将用户页面重定向到Portal Server的认证页面。一般来说，实现手段有两种，一种是设备仿冒服务器向用户发送一个HTTP 302重定向报文，将这个用户重定向到Portal服务认证页面，另外一种是设备仿冒服务器向用户发送一个HTTP 200 OK的报文，但是在HTTP报文体中加入javascript脚本，通过浏览器执行这个脚本来将用户重定向到Portal服务认证页面。如图2所示。

图2 通过javascript脚本重定向

浏览器收到设备仿冒服务器回应的报文，继而向Portal服务器发起请求，Portal服务器会在回应报文中下发一个Cookie，这个Cookie在服务器端唯一标识了这个认证用户。浏览器则会在之后所有的请求报文中携带这个Cookie值。

浏览器在与服务器多次交互的过程中还会获得一些其他的认证要素，比如userip、userurl等。

1.2 认证上线

浏览器获取到认证页面后，用户填入用户名和密码进行身份验证。上线过程主要是由两个HTTP POST请求实现。

图3 用户上线请求报文

如图3所示，真正用作认证的是HTTP BODY字段的信息，userName、userPwd分别对应用户输入的用户名和密码，Cookie值、userip、browserFinalUrl等是之前从服务器获取的值外加一些用户信息，而语言等信息则是来自用户侧主机。

如果认证成功，服务器就会返回部分参数，包括语言、心跳间隔、心跳超时时间、用户上线端口等。

之后浏览器再次发送一个HTTP POST请求，将服务器返回的参数进行组合并发回服务器，至此为一个完整的用户上线过程。

1.3 用户下线

用户下线的过程就简单多了，浏览器还是通过一个HTTP POST完成，请求中除了携带Cookie字段之外，BODY中还携带了language、userip以及serialNo三个值，服务器端根据这些认证要素找到唯一对应的在线用户，完成用户下线工作。

2 测试组网图

测试组网图见图4。

图4 测试组网图

其中，应用层测试仪为Spirent Avalanche 2900，被测设备DUT启用三层Portal认证，这里采用的是SecBlade IAG，Portal服务器和AAA服务器选用iMC V5。测试仪的PORT1模拟用户，PORT2模拟用户想要访问的HTTP Server。

为保证完成测试项目，测试仪需满足以下要求：

Ø 完整支持HTTP协议；

Ø 支持提取服务器端返回的HTTP报文头或者报文体中的内容；

Ø 支持对提取出来的内容进行重组，并且添加到HTTP请求报文中；

Ø 支持以列表形式存储用户名和密码，并且按顺序进行调用。

考虑到测试仪发出的密码为实际密码加密之后的值，因此在向iMC中导入不同用户名的用户时，密码应设置为相同的值，以提高测试的可操作性。

3 测试过程

一个完整的测试流程为：

1. 测试仪模拟用户通过浏览器访问HTTP Server；

2. DUT将未认证的用户重定向到iMC Server进行认证；

3. 测试仪模拟客户端上传用户名、密码进行Web认证；

4. 通过认证后用户再次访问HTTP Server；

5. 访问成功后用户下线。

在整个测试过程中和结束后可以通过PORT2的统计信息来推算出认证速率，因为只有认证成功的用户才能对HTTP Server进行访问。

3.1 测试仪模拟用户上线

由于各种测试仪器配置方法不同，本文仅以Avalanche为例进行说明：

1. 建立一个用户名数据库。勾选Auto Increment，在测试过程中测试仪将会从中逐一获取用户名进行认证。如图5所示。

图5 创建用户名数据库

2. Search Criteria功能。在认证过程中，许多种浏览器发往Portal服务器的认证要素都是从服务器之前回应的报文中取得的，在测试中通过Search Criteria功能完成从服务器回应报文中获取认证要素的工作。获取重定向URL的设置见图6。

图6 获取重定向URL

3. 选择测试仪模拟的浏览器类型，比如IE7等。无论选择哪种浏览器，一定要勾选上Persistence，如图7所示。

图7 浏览器版本选择

此外还需要打开重定向和Cookie功能，如图8所示。

图8 开启重定向及Cookie功能

4. Actions的配置。这是整个测试过程中最重要的环节，通过一条一条的脚本模拟用户上线的每一个动作。

（1）首先模拟用户访问一个HTTP Server；

（2）检查回应报文的类型，如果是HTTP 302重定向报文则根据报文内容进行重定向，如果是HTTP 200 OK，则获取报文体中的javascript脚本，并根据脚本内容访问指定的Portal认证页面；

以上两步旨在模拟强制认证情况下，被测设备的重定向性能，如果不需要关注该项性能指标，可以直接访问Portal认证页面模拟用户主动认证的情况。

（3）继续检查回应的HTTP报文，从中获取例如Cookie、用户IP、访问IP等认证要素；

（4）从用户名数据库中选取一个用户名，并且加入之前获取的认证要素组成认证请求，发送至Portal服务器；

（5）继续检查回应的HTTP报文，从中获取用户在线信息，并按照规则进行心跳连接；

（6）访问测试仪PORT2模拟的HTTP Server以验证是否达到了正确认证；

（7）使用之前获取的各种认证要素组合成下线请求，完成用户下线，整个上述过程如图9所示。

图9 Avalanche测试脚本

3.2 测试结果分析

所有的配置完成后，便可以通过调整测试仪器的并发用户数量进行Portal的Web认证性能测试。每一个用户必须完成认证之后才能成功访问外网，而每次认证成功之后只访问一次外网服务器，所以在测试过程中和测试结束后可以通过查看测试仪模拟的HTTP Server端成功的新建速率来判定Web认证每秒钟的处理性能。如图10所示，打开Avalanche Run-Time Statistics，选择Server -> Layer 4 -> Connections -> Current connection per second，查看Portal Web认证性能测试结果，图中红色的线表示客户端向HTTP服务器每秒钟发起的请求数量，也就是每秒钟Web认证的成功数量。

图10 Portal Web认证性能测试结果

另外，还可以通过调整测试仪模拟HTTP Server回应的页面大小来控制设备转发负载，从而模拟在真实流量压力下的Web认证性能。

4 测试方法探讨

此外，还可以利用这种思路进行SSL VPN、设备Web管理等基于HTTP和HTTPS的各种模块的性能压力测试，均能达到良好的测试效果。

首先对整个认证过程进行分析，在众多交互报文之中找出认证所必须的报文，一个认证过程往往会有几十甚至上百个报文交互，完全模拟基本是不可能的；然后根据认证流程模拟用户使用密码或者证书进行登录；为了更真实的模拟压力环境，可以在登录成功之后进行一些常见的操作，比如访问几个特定的页面、下载一些特定的文件等；最后完成用户下线操作。

此类测试并不复杂，主要需要深入分析登录流程，并合理利用测试仪器的各项功能。

5 结束语

Web认证的实现千差万别，测试人员可以使用WireShark、HttpWatch等工具对认证过程进行分析，通过找出其中的规律和要点，进而利用测试仪进行大规模性能压力测试，并且根据测试仪输出的报告能够对被测设备的相关性能指标进行更直观的评价。
